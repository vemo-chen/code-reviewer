# 审查目标

- 仅审查本次提交中提供的变更代码，不要臆测未展示的上下文实现。
- 优先识别会影响正确性、稳定性、安全性和可维护性的高价值问题。
- 输出要具体、可执行、可定位，避免泛泛而谈。

# 高优先级问题

- 重点检查空指针、边界条件、异常处理、资源释放、并发访问和事务一致性问题。
- 如发现可能导致功能错误、数据错误、接口失败或线上事故的风险，必须明确指出。

# 安全规则

- 关注 SQL 注入、XSS、命令注入、路径穿越、敏感信息泄漏、鉴权缺失等常见安全问题。
- 关注日志、异常或配置中是否暴露密码、令牌、密钥、手机号、身份证号等敏感数据。

# 可维护性规则

- 关注命名是否清晰、职责是否单一、逻辑是否过深、重复代码是否明显、注释是否缺失或误导。
- 关注是否引入难以测试、难以扩展、难以排查的问题。

# 性能与资源规则

- 关注明显的低效循环、重复查询、不必要的对象创建、大范围锁竞争和潜在内存泄漏。
- 关注数据库访问、网络调用、文件读写是否存在不合理的性能风险。

# 测试与日志要求

- 关注关键分支、异常分支、边界条件是否缺少测试覆盖。
- 关注关键失败路径是否缺少必要日志，以及日志内容是否足够定位问题。

# 输出要求

- 必须返回严格 JSON。
- 只针对当前 diff 中可以确认的问题给出意见，不要输出无法定位的问题。
- 如果没有发现明确问题，可以返回低风险结论，但不要编造问题。
- summary 和 briefSummary 要使用中文。
- comments 中的 filePath 和 line 必须尽量准确，不要虚构。
